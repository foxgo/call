# Call IAM Web SaaS Redesign Design

**日期：** 2026-06-09  
**状态：** 已确认，待实现

## 1. 背景

`call-iam-web` 已具备完整的 IAM 后台信息架构与交互链路，但当前界面更接近可运行原型：

- 视觉语言分散，缺少统一的品牌感与产品感
- 页面、列表、弹窗之间的样式尺度不一致
- 信息层级偏平，缺少成熟 SaaS 控制台的框架感
- 移动端与窄屏布局较弱，密度与操作区域适配不充分

当前目标不是改业务能力，而是在保留已有路由、接口与核心交互的前提下，将 `call-iam-web` 重构为一套现代、简洁、可信的企业级 SaaS 后台。

## 2. 目标与非目标

### 2.1 目标

- 建立统一的 IAM 后台视觉系统
- 将登录页、控制台框架、仪表盘、列表页、树页、审计页、弹窗表单统一到一套设计语言
- 强化信息层级、留白、可读性与状态表达
- 改善桌面端与窄屏响应式表现
- 保留当前业务结构、现有接口和路由能力，避免高风险重写

### 2.2 非目标

- 不调整后端接口
- 不重做业务流程或权限逻辑
- 不引入新的 UI 组件库
- 不把现有页面改造成全新信息架构产品，只做有限的结构优化

## 3. 已确认决策

- 风格方向采用“冷静企业感”
- 视觉气质以浅色背景、低饱和蓝绿色、细边框、高留白为主
- 重构范围覆盖：
  - 登录页
  - 控制台框架
  - 仪表盘
  - 租户、用户、角色、部门、审计五类业务页
  - 相关弹窗表单和交互状态
- 保留现有 Vue 3 + Vite + TypeScript + Pinia + Vue Router 技术栈
- 优先通过共享样式和小范围模板调整完成统一，不进行大规模组件体系重写

## 4. 视觉系统设计

### 4.1 颜色

- 主色：低饱和蓝绿色，表达可信、冷静、企业感
- 中性色：雾灰、浅灰白、深墨色，承担背景、边框、正文、辅助信息
- 语义色：成功、警告、危险、信息全部改为低刺激版本，用于状态 badge、提示和危险操作

### 4.2 层级

整体采用三层结构：

- 背景层：轻渐变与柔和光感，用于建立空间氛围
- 面板层：主要承载页面内容的白色或浅色卡片
- 浮层：弹窗、重点卡片、登录表单等更强的聚焦区域

页面不会依赖浓重阴影营造层级，而是通过边框、圆角、对比度和留白实现更稳定的秩序感。

### 4.3 字体与间距

- 统一标题、正文、辅助文案尺度
- 强化 `12 / 14 / 16 / 20 / 28` 的字号节奏
- 正文以易读性优先，不使用过细字重
- 采用统一的 4/8 像素间距系统，控制卡片、工具栏、表单字段和按钮组之间的节奏

### 4.4 组件气质

- 统一圆角尺度，避免页面和弹窗风格断裂
- 边框细、对比清晰，但不过度强调分割感
- 阴影收敛，只给重要浮层和卡片少量深度
- 按钮、输入框、筛选器、标签全部收敛到一致的视觉语言

## 5. 页面结构策略

### 5.1 登录页

登录页从单卡片表单升级为品牌化入口页：

- 保留单一登录流程
- 增加品牌信息、系统价值表达和环境氛围
- 提高首屏可信感，使其更符合企业 SaaS 产品入口

### 5.2 控制台框架

采用成熟的 SaaS shell：

- 固定侧边导航
- 顶部信息栏承担标题、用户信息和快捷动作
- 内容区统一容器宽度、边距与垂直节奏

### 5.3 仪表盘

仪表盘改为：

- 欢迎/摘要区
- 核心指标卡
- 最近审计活动

核心目标是增强“系统总览”感，而不是仅罗列几个数字。

### 5.4 业务列表页

租户、用户、角色页统一采用：

- 页头
- 工具栏
- 内容面板
- 行内状态与操作区

每条记录分为主体信息区与操作区，减少当前动作挤在一行的压迫感。

### 5.5 部门树页

保留树结构，但强化组织层级表达：

- 更清晰的缩进和分组
- 更像组织管理目录，而不是普通平铺列表
- 操作区与主体信息视觉分离

### 5.6 审计页

审计页将筛选区与结果区分离：

- 筛选区为独立过滤面板
- 列表区强调动作、资源、操作者、时间等关键字段
- 详情弹窗采用事件详情布局，提升可读性

### 5.7 弹窗表单

所有弹窗统一：

- 标题区
- 必要说明或上下文
- 字段分组
- 清晰的底部操作区

复杂字段如权限、部门、角色选择器也保持同一风格。

## 6. 交互与响应式规则

### 6.1 交互状态

- 按钮补齐 `hover / active / disabled / loading`
- 输入框补齐 `focus`
- 卡片和列表项提供轻量悬停反馈
- 危险操作视觉上与普通操作明确区分

### 6.2 动效

- 动效控制在 150–220ms
- 仅用于按钮、卡片、弹窗和小范围状态切换
- 尊重简洁，不做装饰性复杂动画

### 6.3 响应式

- 桌面端使用侧栏布局
- 窄屏时允许侧栏收敛，工具栏换行，操作区下沉
- 保证列表页和弹窗在移动端不会横向溢出

### 6.4 可访问性

- 保留清晰的 focus ring
- 交互元素保持足够点击面积
- 文本与背景对比度优先满足后台可读性
- 不以颜色作为唯一状态表达

## 7. 实现策略

本次重构优先通过以下方式落地：

- 在 `src/styles/index.css` 中建立全局设计令牌和共享 UI 类
- 统一 `ConsoleLayout`、`SideNav`、`HeaderBar` 的框架结构
- 对登录页、仪表盘、业务页、弹窗模板做中等粒度重构
- 尽量复用现有逻辑与测试，只调整模板层次和样式类

## 8. 验证策略

- 运行 `call-iam-web` 现有 `vitest`
- 运行前端构建，确认模板与类型没有回归
- 在浏览器中验证重构后的控制台主框架和关键页面表现
